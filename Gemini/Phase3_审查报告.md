## 2. 核心技术难点质量审查

### 2.1 冰心诀与隐蛊的并发响应架构 (`interrupt.ts`)
- **完美的抽象化设计**：通过统一的 `pendingInterrupts` 记录打断窗口，剥离了具体响应卡牌对核心流转的耦合。
- **动态扫描响应权**：在创建 `createHideBugInterrupt` (隐蛊打断窗口) 时，系统不仅会扫描持有【隐蛊 (`TP03`)】的玩家，还会**自动检测受击者是否装备了【天帝祭服 (`FJ02`)】**。当触发天帝祭服时，允许其将任意手牌当做隐蛊使用。这充分证明了开发者彻底吃透了原版 C# 的规则细节。
- **并发锁与无状态冲突**：通过简单的首个响应者锁定机制（检查 `interrupt.respondedPlayerIds.length > 0` ），成功避免了多人抢出冰心诀时的重复扣除卡牌和状态树混乱问题。

### 2.2 天罡战气的“部分翻倍”逻辑 (`combat.ts`)
- **优雅的纯函数计算**：原版 C# 极易在天罡战气的处理上出现战力池被反复翻倍污染的 Bug。这里的实现极其精妙，在 `calculatePowerWithTianGang` 中采取了每次获取战力时**实时派生**的策略：
  `return (base + equipment + pet + skill) * 2 + combatCard;`
- 不直接在状态树中修改战力基础值，彻底规避了战牌加成被错误翻倍的问题，同时也无需在战后做复杂的还原操作。

### 2.3 特殊装备被动解耦 (`combat.ts`)
- 通过提供诸如 `hasTayunBoots` (踏云靴)、`getDragonArmorReduction`（龙魂战铠减伤）等独立纯读取器（Selectors），使得在任意时刻（如伤害结算、濒死结算）都可以安全且便捷地探查玩家装备被动状态，完全契合纯状态机数据流原则。

## 3. 合规性审查

- **坚持无副作用**：所有特殊牌的触发、检验与卡牌置换都完全是对 `G` 状态树的修改操作，没有直接引入网络包、定时器或 UI 层的耦合。
- **严格的边界控制**：遵循了 Phase 3 的边界约定，只实现了 56 张手牌范围内的能力。例如，在 `engine/combat.ts` 的 `getSkillPower` 方法中，主动留下了 `// TODO: Phase 4 实现技能战力修正` 的存根（Stub），并未越界去提前实现角色技能，展示了极强的项目纪律性。

## 4. 后续建议 (Action Items for Phase 4)

随着卡牌引擎彻底就绪，即将进入工作量最大的 **Phase 4 (角色技能引擎)**，请务必关注以下交接点：

1. **技能注册表（Registry）设计**：在实现 34 个角色多达 70 余个技能时，强烈建议复用类似于当前卡牌的路由分发模式，并引入基于枚举（`SkillTrigger`）的“事件钩子（Hook）机制”，坚决避免在主干流程里写入海量的 `if (heroId === X) else if ...`。
2. **多重变身与状态继承安全**：在实现赵灵儿、魔翳、孔璘等角色的变身机制时，一定要妥善使用预留的 `TransformRelation`。必须确保变身切换英雄模板对象时，玩家的 `currentHp`、`equipment` 和 `hand` 等核心资产能被绝对安全地继承和移植。

# 仙剑逍遥游 Web版 — 统一实施路线

> 本文档综合了 Gemini、Qwen、GLM、Mimo 四份 AI 建议，去重、对齐、取最优方案后形成的唯一指导路线。

---

## 一、技术栈确认

| 层 | 选型 | 说明 |
|---|------|------|
| 引擎 | **boardgame.io** | 游戏状态机、回合管理、多人联机 |
| 前端框架 | **React 18 + TypeScript** | Vite 构建 |
| 样式 | **Tailwind CSS** | 原子化样式 |
| 拖拽交互 | **@dnd-kit** | 手牌拖拽出牌、装备替换 |
| 动画 | **Framer Motion** | 卡牌飞入、翻转、布局过渡 |
| UI 组件 | **shadcn/ui** | Tooltip、Dialog、Toast、Popover |
| 客户端状态 | **zustand** | 纯 UI 状态（hover、选中、面板开关） |
| 工具库 | **nanoid**（短房间号）、**zod**（运行时类型校验）、**clsx + tailwind-merge** | — |

### 不引入的库

| 库 | 原因 |
|---|------|
| Redux | 太重，zustand 完全替代 |
| Socket.io | boardgame.io 自带 WebSocket |
| react-beautiful-dnd | 已停止维护 |
| Ant Design / Material UI | 风格冲突、包体巨大 |
| Three.js / pixi.js | 2D 卡牌不需要 3D 引擎 |

## 二、分阶段实施计划

### Phase 0：准备周（人工为主，3-5天）

**目标**：消除所有模糊信息，产出可执行的数据字典。

**任务清单**：
1. 通读 C# 源码，标注核心类（游戏主循环、战斗结算、技能触发、死亡处理）
2. 导出 psd.db3 数据库为 JSON（Hero/Tux/Monster/Npc/Eve/Skill/Exsp）
3. 对照 C# 源码校准 game-flow.md，消除所有"部分角色"、"敌方推举"等模糊描述
4. 对照 C# 源码校准 card-info.md，补充触发时机、优先级、参数化定义

**产出物**：
- [ ] 7 个 JSON 数据文件（放入 `src/shared/data/`）
- [ ] C# 源码逻辑标注文档
- [ ] 校准后的 game-flow.md
- [ ] 校准后的 card-info.md

**铁律**：Phase 0 没完成，不写一行业务代码。

### Phase 1：数据层（2-3天）

**目标**：定义整个项目的数据骨架。

**任务**：
1. 定义 TypeScript 类型：`StaticHeroDef`、`StaticMonsterDef`、`StaticCardDef`、`CardInstance`
2. 严格区分"静态字典"（不变数据）和"运行时状态"（G 中的数据）
3. 定义 boardgame.io 的 `GameState`（G）接口
4. 枚举定义：阶段、卡牌类型、五行属性、技能触发时机

**产出物**：
- [ ] `src/shared/types/card.ts`
- [ ] `src/shared/types/hero.ts`
- [ ] `src/shared/types/skill.ts`
- [ ] `src/shared/types/game-state.ts`

**铁律**：
- G 必须是纯数据，可 JSON 序列化，绝不能包含类实例或函数
- G 中卡牌只存 instanceId（字符串），不存完整对象

### Phase 2：核心引擎骨架（3-5天）

**目标**：用 boardgame.io 搭建能跑通"一局完整游戏主流程"的最小引擎。**此阶段不实现任何卡牌效果和技能**。

**子步骤**：
1. **项目初始化**：Vite + React + boardgame.io + Tailwind，配置目录结构
2. **主流程骨架**：定义 phases（回合开始→事件→技牌→战斗→补牌→弃牌→回合结束）
3. **Spike 验证**：验证三个关键技术点
   - 顺序响应（A队选支援→B队选妨碍）
   - 动态轮流出牌（战力低者先出，双方交替）
   - 全局打断响应（隐蛊/冰心诀的多人同时响应）

**产出物**：
- [ ] 可运行的 boardgame.io 项目骨架
- [ ] 能通过 Debug Panel 跑通完整回合流程（无实际效果）
- [ ] 三个 Spike 验证结果

**重点**：
- 战斗子流程用 `turn.stages` 或 `activePlayers` 实现
- 横置跳过逻辑（横置角色跳过事件/技牌/战斗/补牌阶段）
- 回合顺序管理（交替落座 A1,B1,A2,B2,...）

---

### Phase 3：卡牌效果引擎（5-7天）

**目标**：实现所有 56 张手牌的效果。

**分四个子批次**：

| 批次 | 内容 | 数量 |
|------|------|------|
| 3.1 | 技牌（鼠儿果、窥测天机、偷盗、铜钱镖、天雷破、五气朝元） | 15张 |
| 3.2 | 战牌（金蝉脱壳、金蚕王、天玄五音、天罡战气） | 17张 |
| 3.3 | 特殊牌（冰心诀、洞冥宝镜、隐蛊、灵葫仙丹） | 14张 |
| 3.4 | 装备（5武器 + 5防具） | 10张 |

**关键难点**：
- 冰心诀的打断窗口机制（暂停当前结算→等待全服响应→超时或收到响应后继续）
- 隐蛊的伤害响应窗口机制
- 天罡战气的"翻倍"计算（翻倍基础+装备+宠物+技能，不翻倍战牌加成）
- 装备替换逻辑（已有同类装备时，新装备替换旧装备）

**产出物**：
- [ ] `src/game/moves/tux-cards.ts`
- [ ] `src/game/moves/combat-cards.ts`
- [ ] `src/game/moves/special-cards.ts`
- [ ] `src/game/moves/equipment.ts`
- [ ] `src/game/engine/combat-power.ts`
- [ ] `src/game/engine/interrupt-window.ts`

### Phase 4：角色技能引擎（5-7天）

**目标**：实现所有 34 个角色的技能。

**分批计划**：

| 批次 | 角色 | 技能数 | 复杂度 |
|------|------|--------|--------|
| 4.1 | 仙剑一（7角色） | 14 | 中（变身机制） |
| 4.2 | 仙剑二（5角色） | 10 | 高（掷骰、变身魔尊） |
| 4.3 | 仙剑三+外传（7角色） | 16 | 高（决斗系统、烹饪/合成） |
| 4.4 | 仙剑四（5角色） | 11 | 高（天河剑判定、赠剑、梦傀儡） |
| 4.5 | 仙剑五（2角色） | 5 | 低 |
| 4.6 | 凤鸣玉誓（8角色） | 16 | 极高（变身、傀儡、雷灵） |

**技能实现规范**：
- 每个技能实现为一个 handler 函数，注册到技能注册表
- 触发时机枚举：TURN_START、EVENT_PHASE、SKILL_PHASE、COMBAT_START、COMBAT_CARD_PHASE、COMBAT_END、ON_HP_CHANGE、ON_DEATH 等
- 强制技能[!]在条件满足时自动触发，可选技能由玩家确认

**产出物**：
- [ ] `src/game/skills/` 目录下按系列的技能实现文件
- [ ] `src/game/skills/registry.ts`（技能注册表）
- [ ] `src/game/skills/transform.ts`（变身系统）

### Phase 6：基础前端（5-7天）

**目标**：用最朴素的 UI 实现完整可玩的游戏。**不要美观，只要能操作**。

**组件清单**：
1. GameBoard（主游戏面板）
2. PlayerPanel（玩家信息面板）
3. HandArea（手牌区）
4. CombatArea（战斗区域）

- 弃牌阶段：手牌超过上限时，点击要弃掉的牌

## 五、核心原则

1. **C# 源码是唯一真相来源**：card-info.md 和 game-flow.md 只作为参考，不作为实现依据
2. **G 是纯数据**：boardgame.io 的 G 必须可序列化，禁止存入函数或类实例
3. **UI 状态与游戏状态分离**：G 只存游戏数据，zustand 只存 UI 交互数据
4. **纯函数**：所有 Moves 必须是纯函数，只能修改 G，禁止 Math.random（用 ctx.random）
5. **严格按 scope.md**：不实现不在范围内的功能，不预留多余接口

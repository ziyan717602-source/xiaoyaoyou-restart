# 仙剑逍遥游 游戏流程

> 整合自：规则书.md、game-rules.md、rules-summary.md

---

## 一、游戏准备

### 分队
- 4-6人，分为A、B两队对抗
- 交替落座（A1, B1, A2, B2, ...），座位决定回合顺序

### 角色选择
- 每人随机发3张角色牌，选1张扣放，其余弃用
- 所有人选完后同时亮出
- 变身牌（赵灵儿·梦蛇、魔尊、龙葵鬼、湮世穹兵）不可直接选，通过条件触发

### 牌堆准备
| 牌堆 | 数量 | 构成 |
|------|------|------|
| 手牌堆 | 56张 | 技牌15 + 战牌17 + 特殊牌14 + 装备10 |
| 事件牌堆 | 14张 | 全部事件牌 |
| 怪物/NPC混合堆 | 30张 | 20怪物 + 10随机NPC（从26张NPC中抽） |

### 初始状态
- 每人从手牌堆顶摸3张起始手牌
- HP = 角色HP上限
- 装备区、宠物区为空

---

## 二、回合流程

每个玩家的完整回合按以下顺序执行：

### 1. 回合开始阶段
- 通常跳过
- 部分角色在此阶段发动技能（如蓄势待发、崩坏）
- **横置处理**：若角色处于横置状态，设置 skipToDiscard 标记，跳过事件/技牌/战斗/补牌阶段

### 2. 事件阶段（高级模式可选）
- 可选择翻取1张事件牌并执行效果，或跳过
- 初级模式下为强制翻取
- 横置状态下强制跳过

### 3. 技牌阶段
可执行以下操作（任意顺序、任意次数）：
- **使用技牌**：绿色牌面的手牌，打出后执行效果
- **放置装备**：蓝色牌面的手牌，放入装备区（每人限1武器+1防具）
- **发动角色技能**：部分技能在此阶段发动
- 横置状态下强制跳过

### 4. 战斗阶段
- **选择打怪**：触发一场战斗（完整流程见下一节）
- **选择放弃**：弃掉怪物堆顶1张牌，跳过战斗
- 醉仙望月步：可触发两次独立战斗
- 横置状态下强制跳过

### 5. 补牌阶段
- 打过怪：补2张手牌
- 跳过战斗：补1张手牌
- 醉仙望月步只打一次：额外多补1张
- 横置状态下强制跳过

### 6. 弃牌阶段
- 手牌上限3张，超过则弃掉多余部分
- 慕容紫英在场时手牌上限改为5张（剑匣技能）
- **横置解除**：弃牌阶段结束后，横置状态清除

### 7. 回合结束阶段
- 通常跳过
- 部分角色在此阶段发动技能

---

## 三、战斗流程（7阶段）

### 阶段1：确定支援妨碍
- **触发者**（当前回合玩家）从己方选择一名**支援者**（不能是自己），可不选
- **妨碍者**由敌方推举一名，可不选

### 阶段2：翻取怪牌 + 命中判定
1. 翻取怪物牌堆顶1张
2. **命中判定**（决定支援/妨碍是否生效）：
   - 触发者：**必然命中**
   - 支援者：`角色总命中 >= 怪物闪避` 则命中，否则失败
   - 妨碍者：`角色总命中 >= 怪物闪避` 则命中，否则失败
3. 翻到NPC牌时（二选一）：
   - 弃掉NPC，继续翻下一张怪物牌
   - 使用NPC效果（见卡牌信息-NPC部分），然后进入补牌阶段补2张牌
4. 可选**混战**：再翻1张牌（每场战斗限1次）
   - NPC：触发方战力+NPC战力（若为倾慕者则x2）
   - 怪物：怪物方战力+新怪物战力，新怪物出场效果和闪避无效

### 阶段3：战斗开始
- 部分角色技能在此阶段发动
- 发动顺序：触发者 → 支援者 → 妨碍者 → 其他角色（按行动顺序）
- 强制发动的战力变化技能在此阶段生效

### 阶段4：出场效果
- 读出怪物牌的【出场效果】并执行
- 若出场效果导致触发者死亡，战斗立刻结束，后续流程跳过

### 阶段5：战牌阶段
1. **初步计算双方战力**：
   - 触发方战力 = 触发者战力（含装备+宠物+技能）+ 支援者战力（若命中）
   - 怪物方战力 = 怪物战力 + 妨碍者战力（若命中）
2. **轮流出战牌**：从战力较低方开始（相等时怪物方先），双方轮流使用战牌
3. **每人每场战斗限用1张战牌**（特殊牌不受此限制）
4. 双方均不再出牌后进入结算

### 阶段6：战斗结算
- **胜利**：触发方战力 >= 怪物方战力 → 执行胜利效果，怪物成为宠物
- **失败**：触发方战力 < 怪物方战力 → 执行失败效果，怪物弃掉
- **金蝉脱壳**：强制结束战斗，无胜败，双方效果均不执行
- 宠物属性唯一性：已有同属性宠物时，需在两只中选择弃掉一只

### 阶段7：战斗结束
- 通常跳过
- 部分角色技能在此阶段发动（如林月如嫉恶如仇、温慧蛮横）

---

## 四、死亡处理流程

当玩家HP降至0时，按以下顺序处理：

### 步骤1：检查自动复活效果
- **五彩霞衣爆发**：弃掉五彩霞衣，复活并恢复2HP
- **蝶精爆发**：弃掉蝶精，满HP复活

### 步骤2：检查灵葫仙丹
- 任意玩家可使用灵葫仙丹令其复活并恢复2HP

### 步骤3：检查倾慕
- 找到所有在场且未使用过倾慕的倾慕者
- 所有倾慕者**同时**各损失1HP
- 目标回复HP = 倾慕者数量
- 倾慕引发的HP损失可能触发链式反应（其他角色的倾慕）
- 倾慕是**强制**的，与队伍无关（敌方角色也会触发）
- 倾慕引发的HP损失**不能用隐蛊免除**
- 每人每局限1次倾慕机会
- 变身角色算同一人（赵灵儿用过倾慕，则梦蛇不能再用）
- 倾慕可能是单向的（如李逍遥→阿奴可触发，但阿奴→李逍遥不可）
- 同时伤害导致倾慕双方均HP=0时，倾慕均无法发动

> **示例**：李逍遥HP=0时，林月如、阿奴在场，二人都要损失1HP，李逍遥回复2HP。若此行为造成林月如HP降至0，则之后李逍遥要损失1HP，林月如回复1HP。

### 步骤4：死亡退场
- 若仍未复活，角色死亡退场
- 弃掉所有手牌、装备、宠物
- 指定1名存活玩家补2张牌（遗言）
- 孔璘死亡时变身为魔尊（装备/手牌/宠物保留）

### 步骤5：变身检查
- 倾慕复活后也检查变身条件
- 赵灵儿：敌方宠物>=3时变身梦蛇（强制）
- 赵灵儿·梦蛇：敌方宠物<3时还原（强制）
- 魔翳：队友全灭时变身湮世穹兵（强制）

---

## 五、混战规则

- 每场战斗限1次，翻到怪物牌后可选择再翻1张
- NPC牌：触发方战力+NPC战力（若该NPC是触发方任意角色的倾慕者，战力x2）
- 怪物牌：怪物方战力+新怪物战力；新怪物出场效果和闪避无效
- 双怪物结算：胜利则两个都入宠物区，失败则依次执行两个失败效果

> **建议使用时机**：(1) 我方战力不足时寄希望于翻出NPC；(2) 我方战力优势巨大时尝试一次获取两个宠物
> **减少风险方法**：(1) 翻怪前用洞冥宝镜查看；(2) 技牌阶段用窥测天机预先安排牌序

---

## 六、横置规则

- 被横置的角色，下回合除弃牌阶段外所有阶段跳过
- 横置不影响当前回合
- 横置状态下依然可以支援、妨碍、使用非回合限定的手牌
- 弃牌阶段结束后横置解除

---

## 七、关键概念

| 概念 | 说明 |
|------|------|
| 在场 | 活着处于游戏中的角色或宠物 |
| 参战 | 触发者、支援者、妨碍者（即使失败也算参战） |
| 爆发 | 装备/宠物满足条件时消耗产生效果，战斗结束前基本数值仍生效 |
| 对调 | 完全对调（非等量），一方无牌也要交出全部 |
| 交换 | 等量交换 |

---

## 八、胜利条件

满足任一即结束：
1. **歼灭胜利**：敌方全部角色死亡退场
2. **宠物战力胜利**：怪物牌堆耗尽时，我方所有宠物战力之和 > 敌方（相等则平局）
3. **认输**：敌方玩家认为无法取胜，中途认输

---

## 九、1对1游戏特别规则

- 每名玩家选择两个游戏角色
- 两名角色共用手牌
- 两名角色HP、装备、宠物不共用
- 可以任意使用两名角色的技能
- 触发战斗时，可以任意选择哪个角色触发，哪个角色支援
- 作为妨碍方时，可以任选一名角色妨碍
- 角色被横置时，两名角色同时横置
- 只要有一个是慕容紫英，弃牌上限即为5张
- 2对2游戏中也可启用此规则（合计8名角色），但触发战斗时不能用自己的另一名角色支援

---

## 十、其他规则

### 双方队伍人数不同
若双方队伍起始人数不一致，请让其中一名玩家同时控制两名角色，手牌分开使用。

### 争人法（替代选角方式）
展示12张角色牌，A、B双方轮流从中剔除一张，然后按以下顺序选择：
1. A队选1名

## 六、数据库结构

数据来源：`../psd48-master/~ex-lib/psd.db3`（SQLite）

### 核心表

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| Hero | 角色数据 | ID, NAME, HP, STR, DEX, SKILL, GENDER, SPOUSE |
| Tux | 手牌数据 | ID, CODE, NAME, COUNT, OCCURS, DESCRIPTION, TARGET |
| Monster | 怪物数据 | ID, CODE, NAME, STR, AGL, LEVEL, SPI |
| Npc | NPC 数据 | ID, CODE, NAME, STR, ACTION |
| Eve | 事件数据 | CODE, NAME, RANGE, EFFECT, SPI |
| Skill | 技能数据 | CODE, TYPE, NAME, OCCURS, PRIORS, DESCRIPE |
| Rune | 符文数据 | CODE, NAME, OCCURS, DESC |
| Exsp | 特殊能力 | TYPE, NAME, HERO, CODE, SKILL, DESC |
| Ops | 操作数据 | CODE, NAME, OCCUR |
| NJ | NPC 行动 | CODE, NAME, INTRO |

### 导出策略

将 psd.db3 导出为 JSON 文件，放入 `src/shared/data/`：
- `heroes.json` - 角色数据
- `tux.json` - 手牌数据
- `monsters.json` - 怪物数据
- `npcs.json` - NPC 数据
- `events.json` - 事件数据
- `skills.json` - 技能数据
- `runes.json` - 符文数据
- `exsp.json` - 特殊能力数据

## 七、联机架构范围

### 7.1 通信协议

- 前端↔后端：WebSocket（JSON 消息）
- 后端内部：单进程，无分布式需求

### 7.2 房间系统

- 创建房间（房主设置人数：2/4）
- 加入房间（输入房间号）
- 房间列表（显示等待中的房间）
- 房主开始游戏

### 7.3 状态同步

- **服务端权威**：所有游戏逻辑在服务端执行
- 客户端只发送操作请求，不执行游戏逻辑
- 服务端广播游戏状态变更给所有客户端

## 八、不做的事（红线）

1. **不翻译不在范围内的卡牌效果** —— 即使代码里有，也不翻译
2. **不引入额外依赖** —— 除非明确必要（如 SQLite 读取库）
3. **不做性能优化** —— 先跑通再优化
4. **不做 UI 美化** —— 先功能后样式
5. **不做多语言** —— 仅中文
6. **不做数据持久化** —— 不保存游戏记录
7. **不做用户系统** —— 无需登录，输入昵称即可

## 六、数据库结构

数据来源：`../psd48-master/~ex-lib/psd.db3`（SQLite）

### 核心表

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| Hero | 角色数据 | ID, NAME, HP, STR, DEX, SKILL, GENDER, SPOUSE |
| Tux | 手牌数据 | ID, CODE, NAME, COUNT, OCCURS, DESCRIPTION, TARGET |
| Monster | 怪物数据 | ID, CODE, NAME, STR, AGL, LEVEL, SPI |
| Npc | NPC 数据 | ID, CODE, NAME, STR, ACTION |
| Eve | 事件数据 | CODE, NAME, RANGE, EFFECT, SPI |
| Skill | 技能数据 | CODE, TYPE, NAME, OCCURS, PRIORS, DESCRIPE |
| Rune | 符文数据 | CODE, NAME, OCCURS, DESC |
| Exsp | 特殊能力 | TYPE, NAME, HERO, CODE, SKILL, DESC |
| Ops | 操作数据 | CODE, NAME, OCCUR |
| NJ | NPC 行动 | CODE, NAME, INTRO |

### 导出策略

将 psd.db3 导出为 JSON 文件，放入 `src/shared/data/`：
- `heroes.json` - 角色数据
- `tux.json` - 手牌数据
- `monsters.json` - 怪物数据
- `npcs.json` - NPC 数据
- `events.json` - 事件数据
- `skills.json` - 技能数据
- `runes.json` - 符文数据
- `exsp.json` - 特殊能力数据

## 七、联机架构范围

### 7.1 通信协议

- 前端↔后端：WebSocket（JSON 消息）
- 后端内部：单进程，无分布式需求

### 7.2 房间系统

- 创建房间（房主设置人数：2/4）
- 加入房间（输入房间号）
- 房间列表（显示等待中的房间）
- 房主开始游戏

### 7.3 状态同步

- **服务端权威**：所有游戏逻辑在服务端执行
- 客户端只发送操作请求，不执行游戏逻辑
- 服务端广播游戏状态变更给所有客户端

## 八、不做的事（红线）

1. **不翻译不在范围内的卡牌效果** —— 即使代码里有，也不翻译
2. **不引入额外依赖** —— 除非明确必要（如 SQLite 读取库）
3. **不做性能优化** —— 先跑通再优化
4. **不做 UI 美化** —— 先功能后样式
5. **不做多语言** —— 仅中文
6. **不做数据持久化** —— 不保存游戏记录
7. **不做用户系统** —— 无需登录，输入昵称即可
